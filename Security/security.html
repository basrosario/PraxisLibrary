Security Philosophy

The site follows four guiding principles:


Least Privilege

Only the minimum required browser capabilities, resource sources, and execution paths are permitted.


Defense in Depth

Multiple, overlapping browser-enforced controls are used so that failure of one mechanism does not result in full compromise.


Predictability

Security policies favor explicit configuration and deterministic behavior, even when this increases implementation effort.


Visibility

Security posture is openly documented and verifiable through public testing platforms, enabling trust through transparency.


Transport Security

All traffic is protected using HTTPS with modern Transport Layer Security (TLS).

  • Plain HTTP is not permitted.
  • TLS configurations disable deprecated protocols and weak cipher suites.
  • Certificates are issued by a trusted public certificate authority.
  • Certificate renewal is automated to reduce operational risk.
  • HTTP Strict Transport Security (HSTS) is enabled with a long max-age and applies to subdomains.

Impact: Prevents protocol downgrade attacks, passive network interception, and certificate-related trust issues.


Browser Security Headers

The site enforces a comprehensive set of HTTP security headers to mitigate common web attack vectors.

  • Cross-site scripting (XSS)
  • Clickjacking and UI redressing
  • MIME type confusion
  • Unintended data leakage via referrer headers
  • Abuse of powerful browser APIs

Header values are intentionally restrictive. Defaults are not relied upon, and permissive wildcard configurations are avoided unless technically justified.

Impact: Shifts significant portions of security enforcement to the browser, reducing reliance on server-side filtering alone.


Content Security Policy (CSP)

The Content Security Policy is the cornerstone of the site‚Äôs security posture. It is designed to:

  • Explicitly define which sources may load scripts, styles, images, fonts, and media
  • Prevent execution of unauthorized or injected code
  • Limit the blast radius of potential third-party compromise
  • Enforce a clear trust boundary for all executable content

Key characteristics include:
  • Explicit allowlists for resource types
  • Avoidance of inline script execution
  • Avoidance of dynamic code evaluation
  • Preference for self-hosted assets over external CDNs

Impact: Significantly reduces the likelihood and impact of client-side injection attacks.


Third-Party and Supply Chain Risk Management

Third-party dependencies are a common source of modern web compromise. This site deliberately minimizes that risk.

  • Limited use of external libraries
  • Preference for self-hosted resources
  • Evaluation of necessity before introducing new dependencies
  • Avoidance of third-party execution contexts where possible

The site does not include analytics, trackers, or advertising scripts that would expand the trust boundary without clear benefit.

Impact: Reduces exposure to upstream compromises and limits external influence over page behavior.


Application and Hosting Controls

The site is designed to expose as little surface area as possible.

  • No administrative interfaces are publicly accessible
  • No authenticated user workflows exist
  • Static content is favored where feasible
  • Hosting platform defaults are reviewed and hardened
  • Services operate under the principle of least privilege

Impact: Limits opportunities for abuse and reduces the consequences of misconfiguration.


Testing and Validation

Security posture is validated using non-intrusive, publicly accessible analysis tools that evaluate:

  • TLS and certificate configuration
  • HTTP security headers
  • Browser enforcement behavior

Testing is limited to the public-facing surface of the site and does not involve active exploitation or penetration techniques.

Impact: Provides ongoing assurance without introducing legal, ethical, or operational risk.

Security Implementation Highlights

Content Security Policy (CSP)

Strict CSP implementation prevents XSS attacks by controlling which resources can be loaded. Default-src set to 'none' with explicit allowlists for each resource type.

HTTP Strict Transport Security

HSTS enforces HTTPS-only connections for 1 year, including all subdomains, preventing protocol downgrade attacks and cookie hijacking.

X-Frame-Options

Set to DENY to prevent clickjacking attacks by blocking the site from being embedded in iframes on other domains.

Permissions Policy

Restricts browser features and APIs, allowing only clipboard-write for copy functionality while blocking camera, microphone, geolocation, and payment features.

Maintenance and Change Awareness

Security is treated as a continuous process rather than a one-time configuration.

  • Security posture is reviewed periodically
  • Reviews are performed after infrastructure, hosting, or configuration changes
  • Configuration drift is actively avoided
  • Security regressions are treated as defects

Impact: Ensures long-term consistency and prevents silent degradation of controls.


Scope and Limitations

An intentionally hardened public surface does not imply immunity from all threats. Out of scope risks include:

  • Credential compromise via unrelated systems
  • Social engineering
  • Zero-day vulnerabilities in browsers or hosting platforms
  • Compromise of external dependencies beyond the site‚Äôs control

These limitations are acknowledged and accepted as part of a risk-informed design.
