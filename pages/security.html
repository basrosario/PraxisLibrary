Our Security Philosophy

Intentional hardening from day one, not an afterthought

Security as a Practice

Security isn't something you do once and forget. It's a continuous practice woven into every decision, every line of code, and every feature we build. Praxis was designed with security as a core principle, not a bolt-on addition.

Defense in Depth

Multiple layers of security controls ensure that if one layer fails, others remain. We don't rely on a single security measure but implement overlapping protections at every level.

Principle of Least Privilege

Our Content Security Policy allows only what's absolutely necessary. No inline scripts, no external resources, no unnecessary permissions. If it's not needed, it's blocked.

Zero Trust Architecture

We assume no external resource is trustworthy. All styles, scripts, and assets are self-hosted. No CDNs, no external APIs, no third-party tracking or analytics.

Continuous Improvement

Every code change is reviewed against security standards. We regularly audit for vulnerabilities, update our practices, and incorporate new security research into our approach.

Content Security Policy (A+ Rating)

Our CSP is one of the most restrictive you'll find on the web

Content Security Policy (CSP) is a browser security mechanism that helps prevent cross-site scripting (XSS), clickjacking, and other code injection attacks. Our policy achieves a perfect A+ rating by being intentionally restrictive.

default-src 'none'

Block everything by default. Every resource type must be explicitly allowed. This prevents any unexpected content from loading.

script-src 'self'

Only scripts from our own domain are allowed. No inline scripts, no eval(), no external JavaScript. This eliminates XSS attack vectors.

style-src 'self'

Only stylesheets from our domain. No inline styles means no CSS injection attacks. All styling is in our single external stylesheet.

connect-src 'self'

Network requests only to our own domain. No external API calls, no tracking beacons, no data exfiltration possible.

form-action 'none'

Forms cannot submit data anywhere. This prevents form hijacking attacks where malicious actors redirect form submissions.

base-uri 'none'

The base URL cannot be changed. This prevents base tag injection attacks that could redirect all relative URLs.

Why This Matters

Most websites allow inline scripts and styles for convenience. We chose the harder path: refactoring all code to work with strict CSP. The result is a site that's practically immune to XSS attacks, the most common web vulnerability.

OWASP Security Standards

Protecting against the most critical web application security risks

The OWASP (Open Web Application Security Project) Top 10 represents the most critical security risks to web applications. Here's how Praxis addresses each one:

A01: Broken Access Control

Status: N/A - Praxis has no user accounts, no authentication, no access control. There's nothing to break because there's nothing to protect at the user level.

A02: Cryptographic Failures

Status: Protected - No sensitive data is collected, stored, or transmitted. No encryption needed because there's no sensitive data to encrypt.

A03: Injection

Status: Protected - No database, no server-side code execution, strict CSP prevents script injection. All content is static and pre-rendered.

A04: Insecure Design

Status: Protected - Security was designed in from the start. Defense-in-depth architecture, not security as an afterthought.

A05: Security Misconfiguration

Status: Protected - Minimal attack surface. No admin panels, no configuration files, no unnecessary features enabled.

A06: Vulnerable Components

Status: Protected - Zero external dependencies. No npm packages, no CDN libraries, no third-party code that could contain vulnerabilities.

A07: Auth Failures

Status: N/A - No authentication system means no authentication vulnerabilities. Simplicity is security.

A08: Data Integrity

Status: Protected - All code is self-hosted. No external scripts that could be compromised. CSP prevents unauthorized code execution.

A09: Logging Failures

Status: N/A - No logging needed because there are no security events to log. No user actions are tracked or recorded.

A10: SSRF

Status: N/A - No server-side code means no server-side request forgery. All processing happens in the user's browser.

Privacy by Design

We can't leak what we don't collect

Privacy isn't just about compliance; it's about respect for our users. We made deliberate choices to protect your privacy:

No Analytics

No Google Analytics, no tracking pixels, no session recording. We don't know who you are, where you're from, or what you're doing. And we like it that way.

No Cookies

Praxis sets no cookies. Your browser preferences (like accessibility settings) are stored only in localStorage on your device, never sent to any server.

No External Requests

Zero requests to external servers. No fonts from Google, no scripts from CDNs, no images from third parties. Your browser only talks to our server.

Client-Side Processing

All tools (Prompt Analyzer, Method Matcher, etc.) run entirely in your browser. Your prompts and data never leave your device.

Verify It Yourself

Open your browser's developer tools (F12), go to the Network tab, and reload this page. You'll see only requests to praxis's own domain. No third parties. No trackers. Complete transparency.

Continuous Security Auditing

Security isn't a destination, it's a journey

Every code change goes through security review. Here's what we check:

CSP Compliance

No inline styles or scripts. All styles in styles.css, all scripts in app.js. Every PR is verified against CSP rules.

XSS Prevention

innerHTML usage is reviewed for safe patterns. User input is never directly rendered. Template literals use safe content only.

Dependency Audit

Zero external dependencies means zero supply chain risk. No package.json, no node_modules, no npm audit needed.

Link Validation

All external links use rel="noopener noreferrer". Internal links are verified to exist. No broken references.

Code Quality

Consistent patterns, proper error handling, no eval() or dynamic code execution. Clean, readable, auditable code.

Performance Review

Security and performance go hand in hand. No render-blocking resources, efficient code, fast load times reduce attack surface.
