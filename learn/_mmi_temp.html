Max Mutual Information

Selecting demonstrations that create the strongest information bridge between question and correct answer — optimizing examples through information theory to maximize what the model learns from each one.

Framework Context: 2022

Introduced: Published by Sorensen et al., Max Mutual Information uses information theory to select which few-shot examples maximize the model’s ability to predict correct outputs. Instead of picking examples by surface similarity, MMI chooses examples that make the correct answer most probable. The approach treats demonstration selection as an optimization problem — measuring the mutual information between the prompt (with candidate examples) and the target output, then selecting the combination that scores highest.

Modern LLM Status: While modern LLMs are less sensitive to example selection, the principle remains valuable in automated pipelines. The information-theoretic perspective has influenced how prompt optimization tools evaluate demonstration quality. In production systems that process thousands of queries, even small gains from better example selection compound into meaningful accuracy improvements. MMI’s core insight — that the best examples are those that make correct outputs most predictable — continues to guide research in automated prompt engineering.

The Core Insight

Not All Examples Are Created Equal

When building a few-shot prompt, the instinct is to grab examples that look similar to the input query. But surface similarity is a crude proxy for what actually matters: how much does this example help the model produce the correct output? Two examples might look equally relevant by keyword overlap, yet one dramatically improves accuracy while the other adds noise or confusion.

MMI finds the examples that create the strongest information bridge. Rather than asking “which examples resemble the input?” it asks “which examples make the correct answer most predictable?” This reframes demonstration selection from a retrieval problem into an optimization problem — scoring each candidate by its measurable impact on output probability.

Think of it like choosing study materials for an exam: instead of reading everything related to the topic, you focus on the specific practice problems that most clarify the patterns you need to recognize on test day.

Mutual Information, Intuitively

Mutual information measures how much knowing one thing tells you about another. In MMI, we measure how much seeing a particular example tells the model about what the correct output should be. High mutual information means the example makes the right answer highly predictable. Low mutual information means the example is noise — it does not help the model distinguish the correct output from incorrect alternatives.

The MMI Selection Process

Four stages from candidate pool to optimized prompt

1

Candidate Pool

Assemble potential few-shot examples from available data. This pool might come from a training set, a curated knowledge base, or historical query-response pairs. The larger and more diverse the pool, the more room MMI has to find high-value demonstrations.

Example

For a sentiment classification task, gather 500 labeled product reviews spanning positive, negative, and neutral categories across multiple product types.

2

Score Each Candidate

Measure how much each candidate example increases the probability of the correct output. For each potential demonstration, construct a prompt that includes it, then evaluate how confident the model becomes about the right answer. The score reflects the information gain — how much uncertainty about the correct output is reduced by including this example.

Example

Review A increases P(positive) for a positive test case from 0.6 to 0.92. Review B only increases it from 0.6 to 0.68. Review A scores much higher under MMI.

3

Greedy Selection

Pick the highest-scoring examples iteratively. Start with the single best example, then find the next example that provides the most additional information gain when combined with the first. Repeat until you reach the desired number of demonstrations. This greedy approach avoids redundancy — each new example is selected for what it adds beyond what is already included.

Example

After selecting a strongly positive electronics review, the algorithm next picks a subtly negative restaurant review — it adds the most new information about the classification boundary.

4

Prompt Assembly

Combine the selected examples into the final optimized prompt. The order and formatting of demonstrations matter — arrange them to build a clear pattern the model can follow. The result is a prompt where every example earns its place through measurable contribution to output quality.

Example

The final 5-shot prompt contains examples specifically chosen to cover the classification boundary, handle edge cases, and maximize the model’s confidence on the target query.

See the Difference

Why information-theoretic selection outperforms naive approaches

Random/Similarity Selection

Approach

Pick examples randomly from the available pool or select the most surface-similar examples using keyword overlap or embedding distance. No measurement of actual impact on model output.

Result

Hit-or-miss quality. Some examples may help, others add noise or even mislead the model. Redundant examples waste prompt space. No guarantee that selected demonstrations improve accuracy on the target task.

Unoptimized, inconsistent, no feedback on example quality
VS

MMI Selection

Approach

Score each candidate example by how much it increases the model’s probability of producing the correct output. Select examples greedily to maximize cumulative information gain. Every demonstration is chosen for measurable impact.

Result

Systematic optimization. Each example earns its place in the prompt through quantified contribution. Redundancy is minimized because the greedy process favors examples that add new information. Accuracy improvements are consistent and measurable.

Optimized, consistent, every example justified by output probability

Self-Ask in Action

See how explicit sub-questions improve multi-hop reasoning

Original Question

“Was the iPhone invented in the same state where President Gerald Ford was born?”

Self-Ask Decomposition

Are follow-up questions needed here? Yes.

Follow-up: Where was the iPhone invented?
Intermediate answer: The iPhone was developed by Apple Inc. in Cupertino, California.

Follow-up: Where was President Gerald Ford born?
Intermediate answer: Gerald Ford was born in Omaha, Nebraska.

Follow-up: Are California and Nebraska the same state?
Intermediate answer: No, California and Nebraska are different states.

So the final answer is: No. The iPhone was invented in California, while Gerald Ford was born in Nebraska.

Original Question

“Did the treaty that ended World War I contribute to the conditions that started World War II?”

Self-Ask Decomposition

Are follow-up questions needed here? Yes.

Follow-up: What treaty ended World War I?
Intermediate answer: The Treaty of Versailles, signed in 1919.

Follow-up: What were the key provisions of the Treaty of Versailles?
Intermediate answer: It imposed heavy reparations on Germany, required territorial concessions, limited Germany’s military, and assigned war guilt to Germany (Article 231).

Follow-up: How did these provisions affect Germany in the 1920s and 1930s?
Intermediate answer: The reparations contributed to economic instability, hyperinflation, and widespread resentment. These conditions fueled nationalist movements, including the rise of the Nazi Party.

Follow-up: Did these conditions contribute to the start of World War II?
Intermediate answer: Yes. Historians broadly agree that the punitive terms of Versailles created political and economic conditions that Adolf Hitler exploited to gain power and pursue aggressive expansion.

So the final answer is: Yes. The Treaty of Versailles imposed conditions on Germany that contributed to the economic hardship and political instability from which World War II emerged, though it was one of several contributing factors rather than the sole cause.

Original Question

“Was the element gallium discovered before or after the scientist who predicted its existence died?”

Self-Ask Decomposition

Are follow-up questions needed here? Yes.

Follow-up: Who predicted the existence of gallium?
Intermediate answer: Dmitri Mendeleev predicted gallium’s existence in 1871, calling it “eka-aluminium” based on gaps in his periodic table.

Follow-up: When was gallium actually discovered?
Intermediate answer: Gallium was discovered in 1875 by Paul Emile Lecoq de Boisbaudran.

Follow-up: When did Dmitri Mendeleev die?
Intermediate answer: Mendeleev died on February 2, 1907.

Follow-up: Was 1875 before 1907?
Intermediate answer: Yes, 1875 is 32 years before 1907.

So the final answer is: Gallium was discovered before Mendeleev died. He predicted it in 1871, it was found in 1875, and he lived until 1907 — meaning he witnessed the confirmation of his prediction during his lifetime.

When to Use Self-Ask

Best for multi-hop questions that demand transparent reasoning

Perfect For

Multi-Hop Questions

Questions that chain multiple facts together — “Who was the president when the company founded by the inventor of the telephone went public?”

Reasoning Audits

When you need to verify every step of the model’s logic — each sub-question and answer creates a checkable evidence trail.

Educational Contexts

Teaching students how complex questions decompose — Self-Ask makes the hidden reasoning structure visible and instructive.

Fact-Checking Pipelines

Breaking claims into verifiable sub-claims — each intermediate answer can be independently confirmed against trusted sources.

Skip It When

Single-Hop Questions

Questions answerable with a single fact lookup — “What is the capital of France?” needs no decomposition.

Speed-Critical Applications

When latency matters more than transparency — Self-Ask adds tokens and processing time for each sub-question and answer pair.

Creative or Open-Ended Tasks

Writing, brainstorming, or opinion-based questions where there is no factual chain to decompose — Self-Ask is designed for factual multi-hop reasoning.

Use Cases

Where Self-Ask delivers the most value

Research Analysis

Decompose complex research questions into verifiable sub-inquiries, ensuring each factual claim is independently supported before drawing conclusions.

Legal Due Diligence

Break regulatory compliance questions into jurisdiction-specific sub-questions, tracing each requirement back to its statutory source.

Medical Reasoning

Decompose differential diagnosis questions into symptom-by-symptom analysis, making each diagnostic inference explicit and reviewable by clinicians.

Customer Support Escalation

When a support issue spans multiple systems or policies, decompose the customer’s problem into sub-questions that each map to a specific knowledge base article.

Security Incident Analysis

Trace attack vectors through explicit sub-questions: what vulnerability was exploited, what systems were affected, what data was accessed, and what was the blast radius.

Financial Analysis

Break investment thesis questions into sub-analyses: market conditions, company fundamentals, competitive positioning, and risk factors — each answered and sourced independently.

Combine with Search

Self-Ask was originally designed to pair each sub-question with a search engine lookup — making it a precursor to modern retrieval-augmented generation (RAG). In production systems, you can route each “Follow-up question” to a knowledge base or API, then feed the retrieved answer back as the “Intermediate answer” for maximum factual grounding.
